# Multi-node — Руководство пользователя

Запустите один control plane Raven перед несколькими нодами Xray, за единым
набором URL подписки. Пользователи автоматически распределяются по нодам, а
каждая подписка позволяет клиенту раскладывать трафик между ними и переключаться
при отказе.

Это руководство для **операторов**, запускающих Raven-subscribe. Про архитектуру
и обоснование решений см. [multi-node-design.ru.md](multi-node-design.ru.md).

> **Одиночная нода не затрагивается.** Если в конфиге нет секции `nodes`, ничего
> из этого не применяется, и поведение побайтово идентично прежнему. Беритесь за
> multi-node только тогда, когда вам действительно нужно несколько входных
> серверов.

---

## Что это вам даёт

- **Более широкая входная поверхность.** Подписка каждого пользователя несёт по
  одному аутбаунду на ноду, под общим балансировщиком. Клиент раскладывает
  трафик между нодами, что разнообразит входные IP и ASN, которые раскрывает один
  токен, — это и есть свойство, важнее всего противостоящее цензуре, блокирующей
  по IP назначения.
- **Автоматическое переключение (failover).** Клиент выбирает лучшую ноду
  (наименьший пинг / наименьшая нагрузка) и сам уходит с мёртвой. Никакого
  переключения на стороне сервера.
- **Масштабирование без дублирования.** Добавляйте мощности или локации, не
  поднимая второй Raven и вторую базу пользователей.

## Понятия

| Термин | Значение |
|---|---|
| **Control plane** | Ваш экземпляр Raven-subscribe — единственный источник истины по пользователям. |
| **Нода** | Удалённый сервер Xray, настроенный идентично остальным. |
| **Однородные** | Все ноды разделяют одни и те же инбаунды, ключи REALITY и SNI. Они различаются только тем, куда подключаются клиенты (`public_host:public_port`) и куда Raven достукивается до них (`api_addr`). Благодаря этому один конфиг клиента работает на **любой** ноде. |
| **Размещение** | С каких нод обслуживается конкретный пользователь. |

## Прежде чем начать

Каждую ноду нужно настроить **до** того, как Raven сможет её использовать:

- Xray работает с **тем же** конфигом инбаундов / REALITY, что и остальные ваши
  ноды. Ansible-роль `role_xray_node` делает это — см.
  [INSTALL §6](INSTALL.ru.md) — либо настройте вручную.
- gRPC API Xray доступен для control plane через WireGuard (рекомендуется) или
  mTLS (шаг 2).
- Raven **не** устанавливает и не настраивает Xray. Он лишь распределяет
  пользователей по нодам, которые уже подняты и настроены идентично.

## 1. Объявите свои ноды

Добавьте массив `nodes` в `config.json`:

```jsonc
{
  "server_host": "vpn.example.com",
  "admin_token": "your-secret-token",
  "base_url": "https://vpn.example.com",

  "nodes": [
    {
      "name": "eu-1",                    // стабильный id ноды (используется как ключ в БД)
      "api_addr": "10.7.0.1:10085",      // gRPC API Xray — приватный/WireGuard-адрес (шаг 2)
      "inbound_tag": "vless-reality-in", // инбаунд, в который добавляются пользователи
      "public_host": "eu1.example.com",  // куда подключаются клиенты для этой ноды
      "public_port": 443,
      "enabled": true                    // в ротации; если опустить → true
    },
    {
      "name": "eu-2", "api_addr": "10.7.0.2:10085",
      "inbound_tag": "vless-reality-in",
      "public_host": "eu2.example.com", "public_port": 443
    }
  ]
}
```

| Поле | Обязательно | Описание |
|---|---|---|
| `name` | да | Стабильный уникальный id ноды. |
| `api_addr` | да | `host:port` gRPC API Xray этой ноды. Должен быть приватным/WireGuard-адресом, если не защищён через `tls` или `allow_public_grpc`. |
| `inbound_tag` | да | Тег инбаунда, в который добавляются пользователи на этой ноде. |
| `public_host` / `public_port` | да | Куда подключаются клиенты для этой ноды. |
| `enabled` | нет | `false` выводит ноду из ротации (по умолчанию `true`). |
| `tls` | нет | mTLS для gRPC-соединения с этой нодой — см. шаг 2. |
| `allow_public_grpc` | нет | Подтверждает публичный `api_addr`, защищённый иным способом (например, внешним mTLS-сайдкаром). |

Когда присутствует `nodes`, устаревшие поля одиночной ноды (`xray_api_addr`,
`api_user_inbound_tag`) больше не используются для провижининга; `server_host`
по-прежнему служит хостом по умолчанию для балансировщика.

## 2. Защитите управляющий канал

gRPC API ноды — это **полный контроль** над этой нодой. Он никогда не должен быть
доступен в открытом виде на публичном адресе — Raven откажется стартовать, если
это так. Два варианта:

**WireGuard (рекомендуется).** Разместите API Xray ноды на WireGuard-адресе и
укажите этот адрес как `api_addr`. Шифрование и взаимная аутентификация
обеспечиваются WireGuard; сертификатов, которыми нужно управлять, нет. Приватный
`api_addr` не требует дополнительной настройки.

**mTLS (для нод без WireGuard).** Добавьте на ноду блок `tls`. Raven предъявляет
клиентский сертификат и проверяет ноду по вашему CA:

```jsonc
{
  "name": "eu-3",
  "api_addr": "203.0.113.7:10085",       // публичный адрес допустим — защищён через tls ниже
  "inbound_tag": "vless-reality-in",
  "public_host": "eu3.example.com", "public_port": 443,
  "tls": {
    "ca_cert":     "/etc/raven/pki/node-ca.pem",     // CA, подписавший серверный сертификат ноды
    "client_cert": "/etc/raven/pki/raven-client.pem",
    "client_key":  "/etc/raven/pki/raven-client.key",
    "server_name": "eu-3.internal"                   // опционально; по умолчанию — хост из api_addr
  }
}
```

Сертификаты читаются один раз при старте; отсутствующий или нечитаемый сертификат
останавливает Raven, а не приводит к молчаливому откату на открытый текст.
Сторона ноды (серверный сертификат + `clientAuth`) настраивается её
Ansible-ролью.

## 3. Разместите пользователей на нодах

По умолчанию новый пользователь размещается на **всех включённых нодах**. Чтобы
закрепить пользователя за подмножеством или изменить размещение позже,
используйте admin API (заголовок `X-Admin-Token`):

```bash
# Создать пользователя на конкретных нодах (опустить "nodes" → все включённые ноды)
curl -H "X-Admin-Token: $TOKEN" -X POST https://vpn.example.com/api/users \
  -d '{"username":"alice@vpn.example.com","nodes":["eu-1","eu-2"]}'

# Просмотреть / заменить / убрать размещение пользователя
curl -H "X-Admin-Token: $TOKEN"          https://vpn.example.com/api/users/alice@vpn.example.com/nodes
curl -H "X-Admin-Token: $TOKEN" -X POST  https://vpn.example.com/api/users/alice@vpn.example.com/nodes -d '{"nodes":["eu-1"]}'
curl -H "X-Admin-Token: $TOKEN" -X DELETE https://vpn.example.com/api/users/alice@vpn.example.com/nodes/eu-2

# Список настроенных нод (только чтение — ноды управляются в конфиге, а не создаются через API)
curl -H "X-Admin-Token: $TOKEN" https://vpn.example.com/api/nodes
```

После этого подписка пользователя содержит по одному сбалансированному аутбаунду
на каждую размещённую ноду.

## 4. Проверьте здоровье нод

```bash
curl -H "X-Admin-Token: $TOKEN" https://vpn.example.com/api/sync/status
```

Ответ несёт привычные поля одиночной ноды плюс карту `nodes`:

```jsonc
"nodes": {
  "eu-1": { "reachable": true,  "users_target": 42, "users_present": 42, "last_apply_ok": true },
  "eu-2": { "reachable": false, "users_target": 42, "users_present": 0,  "last_error": "..." }
}
```

- `reachable` — последнее gRPC-соединение с нодой прошло успешно.
- `users_target` против `users_present` — сколько пользователей должно быть на
  ноде (по данным БД) против того, сколько она сейчас сообщает.

## 5. Операции Day-2

- **Добавить ноду.** Настройте её (шаг *Прежде чем начать*), добавьте в массив
  `nodes` и перезапустите Raven. Существующие пользователи будут дозаписаны на
  новую ноду при следующей сверке (reconcile).
- **Вывести ноду из ротации.** Установите `"enabled": false` (или удалите её из
  конфига — тогда она помечается отключённой, а не удаляется, так что её
  размещения сохранятся, если вы вернёте ноду). Клиенты автоматически исключают
  её через балансировщик.
- **После перезапуска ноды не делайте ничего.** Xray держит добавленных через API
  пользователей в памяти, поэтому перезапуск ноды их сбрасывает — но цикл
  reconcile заново применяет пользователей каждой ноды на каждом интервале
  синхронизации, так что нода самовосстанавливается в течение одного интервала.

## Устранение неполадок

| Симптом | Вероятная причина / решение |
|---|---|
| Нода показывает `reachable: false` | Неверный `api_addr`, упавший маршрут WireGuard или файрвол. Для mTLS проверьте, что сертификаты совпадают. Цикл reconcile повторяет попытку каждый интервал синхронизации. |
| `users_present` меньше `users_target` | Ноду недавно (пере)запустили или она была кратковременно недоступна; следующий проход reconcile её заполнит. Если сохраняется → проверьте логи Xray на ноде. |
| Raven не стартует: *"api_addr … is not a private/loopback address"* | gRPC ноды на публичном адресе без защиты. Используйте WireGuard-адрес, добавьте блок `tls` или установите `allow_public_grpc: true`. |
| Raven не стартует: ошибка mTLS-сертификата | Настроенный `tls`-сертификат отсутствует/нечитаем. Это фатально by design (никакого молчаливого отката на открытый текст) — исправьте путь/права. |

## Ограничения

- **Только однородные ноды.** Все ноды разделяют одни и те же ключи REALITY и
  форму инбаунда. Разные ключи/протоколы на каждой ноде не поддерживаются.
- **Постатистика по нодам не агрегируется для квот.** Трафик пользователя
  разбивается по нодам; суммирование по пользователю — в дорожной карте
  observability.
- **Нет HA для control plane.** Один Raven означает одну базу SQLite.
