# `multi-node` — Дизайн (перевод)

> Каноническая версия этого документа — английская: [multi-node-design.md](multi-node-design.md). Русский перевод может отставать.

Статус: реализовано, вошло в релиз v0.4.0.

Область: Raven-subscribe. Затрагивает `internal/{config,database,api,syncer,xray,core}`.
На вывод подписки для одноузловых установок это **не** влияет (см. §10).
Issue: [#100 multi node support](https://github.com/AlchemyLink/Raven-subscribe/issues/100).
Связано: [`internal-core-design.md`](internal-core-design.md) — multi-node надстраивается над ним.

## 1. Зачем

Обычно один Raven управляет ровно одним Xray: он пишет в локальный `config.d`
и общается с единственным `xray_api_addr`. Чтобы запустить несколько серверов
Xray таким способом, пришлось бы держать рядом с каждым отдельный Raven — со
своей базой SQLite. Это означает дублирование хранилищ пользователей, расходящиеся
токены и несколько мест для администрирования.

Цель (#100): **один Raven владеет всеми пользователями и раздаёт их нескольким
удалённым серверам Xray**, без отдельного Raven на каждый сервер. Для небольших
частных развёртываний это убирает главный операционный барьер для
горизонтального масштабирования.

Изменение оказалось небольшим, потому что gRPC-клиент
(`internal/xray/apiclient.go`) **уже без состояния и рассчитан на несколько
целей**: каждая функция принимает аргумент `apiAddr` и устанавливает соединение
заново. Предположение «один сервер» живёт лишь в нескольких местах (см. §3), а не
на уровне протокола.

## 2. Чем это не является (область)

- **Не второй движок.** Каждая нода — это Xray. sing-box остаётся вне области.
  Multi-node не зависит от абстракции движка.
- **Не разнородные ноды.** Ноды **однородны**: одинаковые инбаунды, ключи
  REALITY, SNI и шифрование VLESS. Различаются они только тем, куда подключаются
  клиенты (`public_host:public_port`) и куда Raven достаёт до них (`api_addr`).
  Это тот же «зеркальный конфиг», что уже используется для RU-моста. Ноды с
  разными протоколами или ключами — отдельный проект.
- **Не инструмент развёртывания.** Raven не устанавливает и не настраивает Xray
  на ноде — это задача Ansible. Raven лишь распределяет пользователей по нодам,
  которые уже подготовлены и настроены одинаково.
- **Не изменение одноузлового пути.** Без секции `nodes` в конфиге поведение
  побайтово идентично прежнему (§10).
- **Не открытый gRPC по интернету.** См. §7 — это было жёстким требованием, а не
  пожеланием.

## 3. Где было зашито «одна нода»

До multi-node предположение об одном сервере жило в четырёх слоях:

- **config** — `XrayAPIAddr` (один gRPC-адрес) и `APIUserInboundTag` (один
  целевой инбаунд).
- **DB-модель** — понятия ноды не было; пользователи отображались прямо на
  инбаунды.
- **api CRUD** — на каждую операцию выбирался один бэкенд:
  `if apiAddr != "" { …gRPC } else { …file }`.
- **syncer** — обнаружение дрейфа читало набор «есть» из локальных файлов
  `config.d`, а здоровье было одним глобальным снимком.

gRPC-клиент (`xray/apiclient.go`) был единственным слоем, который уже работал с
любым числом целей. Multi-node расширяет именно эти четыре слоя и оставляет
gRPC-слой нетронутым.

## 4. Модель ноды

```jsonc
// config.json — новая необязательная секция. Отсутствует → одна нода, ровно как раньше.
{
  "server_host": "eu.example.com",        // legacy: становится неявной локальной нодой
  "xray_api_addr": "127.0.0.1:10085",     // legacy
  "api_user_inbound_tag": "vless-reality-in",

  "nodes": [
    {
      "name": "eu-1",                      // стабильный id ноды (используется как ключ в БД)
      "api_addr": "10.7.0.1:10085",        // gRPC API Xray — на WireGuard-адресе (§7)
      "inbound_tag": "vless-reality-in",   // инбаунд, в который добавляются пользователи
      "public_host": "eu.example.com",     // то, что попадает в аутбаунд клиента
      "public_port": 443,
      "enabled": true,                     // нода в ротации (по умолчанию true)
      "deploy": { "mode": "grpc" }         // необязательно; "grpc" (по умолчанию) | "ssh_rsync" (зарезервировано)
    },
    { "name": "eu-2", "api_addr": "10.7.0.2:10085",
      "inbound_tag": "vless-reality-in", "public_host": "eu2.example.com",
      "public_port": 443 },

    { "name": "eu-3",                      // нода БЕЗ WireGuard → gRPC по публичному адресу, под защитой mTLS
      "api_addr": "203.0.113.7:10085",     // публичный адрес разрешён, потому что присутствует блок tls (§7)
      "inbound_tag": "vless-reality-in", "public_host": "eu3.example.com",
      "public_port": 443,
      "tls": {                             // необязательно; взаимный TLS при gRPC-соединении с этой нодой
        "ca_cert":     "/etc/raven/pki/node-ca.pem",       // CA, подписавший серверный сертификат ноды
        "client_cert": "/etc/raven/pki/raven-client.pem",
        "client_key":  "/etc/raven/pki/raven-client.key",
        "server_name": "eu-3.internal"     // необязательно; по умолчанию — хост из api_addr
      }
    }
  ]
}
```

Правила, применяемые в `config.Load`:

- **Нет `nodes`** → Raven синтезирует одну неявную ноду с именем `local` из
  legacy-полей. Весь нижележащий код всегда работает со списком нод; одна нода —
  это просто случай N=1.
- **`nodes` присутствует** → legacy-поля игнорируются для провижининга
  (`server_host` всё ещё служит запасным хостом для балансировщика).
- **Валидация** — каждое `name` уникально и непусто; `api_addr` непуст; для ноды
  `grpc` адрес должен быть приватным, ЛИБО нести блок `tls`, ЛИБО иметь
  `allow_public_grpc: true` (см. защиту от «выстрела в ногу» в §7). Блок `tls`,
  если присутствует, требует `ca_cert`, `client_cert` и `client_key`.

## 5. DB-модель

Две дополняющие таблицы. Raven-subscribe применяет изменения схемы собственным
идемпотентным `migrate()` (обычный `CREATE TABLE IF NOT EXISTS`) — он **не**
использует goose (goose есть только в дашборде). Существующие одноузловые базы
обновляются прозрачно.

```sql
CREATE TABLE IF NOT EXISTS nodes (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT NOT NULL UNIQUE,
    api_addr    TEXT NOT NULL,
    inbound_tag TEXT NOT NULL,
    public_host TEXT NOT NULL,
    public_port INTEGER NOT NULL,
    enabled     INTEGER NOT NULL DEFAULT 1,
    created_at  TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- какие пользователи размещены на какой ноде. Нет строки = пользователя нет на этой ноде.
CREATE TABLE IF NOT EXISTS user_nodes (
    user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    node_id INTEGER NOT NULL REFERENCES nodes(id) ON DELETE CASCADE,
    PRIMARY KEY (user_id, node_id)
);
CREATE INDEX IF NOT EXISTS idx_user_nodes_node ON user_nodes(node_id);
```

- **`nodes` сверяется с конфигом при старте**: строки upsert-ятся по `name`, а
  нода, исчезнувшая из конфига, помечается `enabled=0`, а не удаляется, так что
  её строки в `user_nodes` сохраняются.
- **Заполнение размещений (backfill)** — пользователь без строк размещения
  помещается на все `enabled`-ноды. При обновлении одноузловой установки это
  кладёт каждого существующего пользователя на `local`, так что поведение не
  меняется. Backfill идемпотентен.

Ключевой момент: блоб учётных данных (`user_clients`) остаётся **на инбаунд**, а
не на ноду. Поскольку ноды однородны, одни учётные данные действительны на любой
ноде, обслуживающей этот `inbound_tag`. `user_nodes` управляет только
*размещением*, но никогда учётными данными — что сохраняет схему компактной.

## 6. Изменения по слоям

### 6.1 `internal/core` — фан-аут как `AdminAPI`

Новый интерфейс не нужен. Multi-node — это **композитная реализация**
существующего `core.AdminAPI`:

```go
// internal/xray/fanout.go
type fanoutAdmin struct {
    targets []namedAdmin // один grpc-admin на ноду
}

func NewFanoutAdmin(nodes []core.NodeTarget) core.AdminAPI { … }

// AddClient на фан-ауте = AddClient на каждой целевой ноде.
// Частичный успех сообщается наверх, но НЕ откатывается (§6.4).
func (f *fanoutAdmin) AddClient(inboundTag, identity string, h core.AddClientHint) (string, error)
```

`AddClient` должен вернуть **одни** учётные данные для сохранения в БД. Поскольку
ноды однородны, учётные данные всюду идентичны, поэтому фан-аут берёт их с первой
успешной ноды и проверяет, что остальные совпадают (несовпадение означает, что
ноды разошлись, и это логируется).

Сигнатуры методов `core.AdminAPI` **не** меняются — фан-аут прячется за тем же
интерфейсом, так что `api.Server` не может отличить одну ноду от N.

Замечание о рефакторинге core: он намеренно заморожен на Фазе 1 — интерфейсы
объявлены, но `internal/api` и `internal/syncer` всё ещё импортируют
`internal/xray` напрямую. Multi-node не ждёт полного рефакторинга (§9); он
перенимает лишь часть `core.AdminAPI`, отвечающую за путь записи.

### 6.2 `internal/api` — CRUD и выбор ноды

- `POST /api/users` принимает необязательное `nodes: ["eu-1","eu-2"]`. Пусто →
  все `enabled`-ноды. Это заполняет `user_nodes`.
- Весь пользовательский CRUD идёт через единственный `s.admin` (фан-аут) — без
  ветвления по бэкенду на каждый вызов.
- Админские эндпоинты: `GET /api/nodes`, а также
  `GET/POST/DELETE /api/users/{id}/nodes` для просмотра и изменения размещения
  пользователя.

### 6.3 `internal/xray/generator.go` — один аутбаунд на ноду

Для каждой ноды, на которую размещён пользователь, генератор выдаёт **один
аутбаунд**, указывающий на `node.public_host:node.public_port` и наследующий
протокол, настройки security и REALITY из `inbound_tag`. Все эти аутбаунды идут
под **существующий балансировщик** (`BalancerStrategy`), так что клиент сам
выбирает живую ноду по least-ping / least-load.

При N=1 (одна нода) аутбаунд побайтово идентичен прежнему (проверяется golden-тестом, §10).

### 6.4 `internal/syncer` — дрейф и статус по каждой ноде

`HandlerService` Xray умеет возвращать живой список пользователей инбаунда по
gRPC (`GetInboundUsers`). Это даёт каждой удалённой ноде **полный reconcile по
сети**: прочитать набор «есть» с ноды, взять набор «нужно» из БД и применить
разницу через `AddUser` / `RemoveUser`. Идея та же, что и у локальной проверки
дрейфа `config.d`, но без чтения файлов с диска.

Цикл reconcile выполняется раз в `SyncInterval` для каждой `enabled`-ноды:

- `AddUser` для уже существующего пользователя возвращает
  `"User X already exists."`, а `RemoveUser` для отсутствующего —
  `"User X not found."`. Фан-аут считает оба безвредными (по совпадению
  сообщения), так что повторное применение того же состояния идемпотентно и не
  засоряет логи ошибками.
- Здоровье ноды = (gRPC доступен) + (число дрейфа после прохода).

**Рестарт ноды стирает пользователей, добавленных через gRPC (см. §12).** Поэтому
цикл reconcile — не только страховка от дрейфа, но и **обязательный механизм
восстановления**: после рестарта удалённой ноды её пользователи пропадают до
следующего прохода, так что нода только на gRPC имеет простой не более одного
`SyncInterval`.

Здоровье сообщается по каждой ноде. `/api/sync/status` возвращает карту `nodes` с
ключами по именам нод, где каждая запись несёт доступность, целевое и фактическое
число пользователей и ошибки применения. Для одной ноды это одна запись, `local`,
так что форма расширяется дополняюще и существующие дашборды продолжают работать.

## 7. Безопасность (жёсткое требование)

`HandlerService` Xray даёт **полный контроль** над пользователями и инбаундами —
а поскольку у него нет контроля доступа на уровне отдельных RPC, то и над
аутбаундами тоже. Выставленный в публичный интернет без аутентификации или TLS,
он позволяет любому, кто дотянется до порта, добавлять и удалять пользователей на
каждой ноде. **Multi-node никогда не должен запускать открытый gRPC по публичной
сети.**

Транспорт выбирается **на каждую ноду, по `api_addr`**, в одном месте —
`dialXrayAPI` / `resolveCredentials`: нода, настроенная с mTLS, соединяется по
TLS, любой другой адрес (WireGuard или loopback) соединяется открытым текстом. Два
варианта, в порядке предпочтения:

1. **gRPC поверх WireGuard (рекомендация по умолчанию).** Привяжите API Xray к
   WireGuard-адресу ноды; шифрование и взаимную аутентификацию обеспечивает
   WireGuard. gRPC остаётся открытым, но недостижим извне туннеля. Это
   вписывается в существующую EU/RU-сеть и не требует сертификатов. Конфиг:
   приватный `api_addr`, без блока `tls`.
2. **mTLS на gRPC (для нод без WireGuard).** Необязательный блок `tls` на ноду:
   `{ca_cert, client_cert, client_key, server_name?}`. Raven выступает
   TLS-клиентом: предъявляет свой клиентский сертификат, проверяет серверный
   сертификат ноды против CA и форсирует TLS 1.3. Сертификаты читаются **один раз
   при старте**, и загрузка **fail-closed** — битый сертификат фатален, никогда
   не тихий откат к открытому тексту на публичный адрес. Ansible-роль ноды
   провижинит соответствующий серверный сертификат и `clientAuth`.

Защита от «выстрела в ногу» в `config.Load`: нода `grpc` с публичным `api_addr` и
без защиты отказывается стартовать. Защита удовлетворяется **любым** из:
приватный/WG-адрес, блок `tls` (mTLS сам по себе — это шифрование +
аутентификация) или явный `allow_public_grpc: true` (запасной выход для
операторов, которые терминируют mTLS внешним сайдкаром).

## 8. Переживание рестарта ноды

Изменения через gRPC живые, но **только в памяти** (§12): они вступают в силу
мгновенно, но теряются при рестарте ноды. Запись пользователей в `config.d` ноды
даёт обратное — надёжно, но применяется только при следующей перезагрузке.
Локальная нода уже получает и то и другое, потому что Raven пишет параллельно в
gRPC и в `config.d` и восстанавливает пользователей при старте.

У удалённой ноды только на gRPC нет локального `config.d`, в который Raven мог бы
писать, поэтому её надёжность обеспечивается циклом reconcile (§6.4): после
рестарта её пользователи восстанавливаются в течение одного `SyncInterval`. Для
небольших установок этого достаточно, поэтому именно это и поставляется. Для
строгой надёжности будущий вариант — также проталкивать данные в `config.d` ноды
по SSH/rsync (`deploy.mode: "ssh_rsync"` зарезервирован в схеме, но пока не
реализован).

## 9. Статус реализации

Поставлено фазами, каждая смерджена отдельным PR (все зелёные, поведение одной
ноды не меняется на каждом шаге) и выпущено в **v0.4.0**:

- **Фаза 0** — принять `core.AdminAPI` для пути записи.
- **Фаза 1** — секция `nodes` в конфиге + таблицы БД, дополняющие и инертные для
  одной ноды.
- **Фаза 2** — `fanoutAdmin` разносит записи по всем нодам.
- **Фаза 3** — аутбаунды на каждую ноду в генерируемых конфигах, под
  балансировщиком.
- **Фаза 4** — API нод/размещения и reconcile + статус по каждой ноде.
- **Фаза 5** — транспорт WireGuard-по-умолчанию и опциональный mTLS; документация
  README/INSTALL; Ansible-роль ноды в Raven-server-install.

## 10. Гарантия: вывод для одной ноды не меняется

Ключевой критерий приёмки. Без `nodes` в конфиге:

- проходят все юнит- и интеграционные тесты;
- `/sub/{token}` и `/c/{token}` (каждое представление) возвращают **побайтово
  идентичный** ответ до и после, на golden-наборе;
- `/api/sync/status` содержит ровно одну запись, `local`.

С однородными `nodes=[A,B]`:

- пользователь, размещённый на обеих, получает балансировщик из двух аутбаундов;
- если B недоступен, пользователь всё равно работает через A, а B показывает
  `reachable:false`;
- публичный `api_addr` без `tls`/`allow_public_grpc` отказывается стартовать; с
  блоком `tls` стартует; битый сертификат фатален (fail-closed).

## 11. Известные ограничения и последующие шаги

- **Политика размещения** — новые пользователи по умолчанию идут на все
  `enabled`-ноды, переопределяется на пользователя. Разные ключи REALITY на ноду
  вне области (это были бы разнородные ноды).
- **Экстренная ротация** — killswitch / ротация инбаундов сейчас нацелены на один
  адрес. Разнести её через `fanoutAdmin` почти бесплатно (он уже покрывает
  `AddInboundFromJSON` / `RemoveInbound`), но это пока не подключено.
- **Статистика по каждой ноде** — `xray-stats-exporter` скрейпит один Xray. При
  балансировке трафик пользователя разбит по нодам, поэтому суммы по пользователю
  нужно складывать по нодам. Это важно для квот/биллинга и относится к дорожной
  карте наблюдаемости.
- **Нет HA для самого Raven** — один control plane означает один SQLite.
  Высокодоступный Raven — отдельный проект.

## 12. Runtime-факты Xray, лежащие в основе дизайна (проверено на `xray-core v1.260327.0`)

Именно эти runtime-поведения Xray делают схему «один control plane → N нод по
gRPC» работоспособной и объясняют, почему некоторые части дизайна устроены именно
так.

- **Изменения пользователей через gRPC живут только в памяти.** Всё состояние
  пользователей хранится в валидаторе в памяти; на диск ничего не пишется. Так
  что рестарт Xray сбрасывает каждого пользователя, добавленного только через
  gRPC — переживают лишь пользователи, присутствовавшие в `config.d` при старте.
  Поэтому цикл reconcile — механизм восстановления, а не просто страховка от
  дрейфа (§6.4, §8).
- **Идемпотентные сообщения об ошибках.** Добавление уже существующего
  пользователя или удаление отсутствующего возвращает конкретное сообщение,
  вместо изменения состояния. Фан-аут полагается на совпадение этих сообщений,
  так что повторное применение того же состояния — no-op.
- **Живой список пользователей читаем.** `GetInboundUsers` возвращает текущих
  пользователей ноды (как gRPC-, так и загруженных из конфига), что и делает
  сетевой reconcile возможным.
- **Ноды делят один trust-домен.** Однородность требует одинаковых ключей REALITY
  на каждой ноде, так что один клиентский конфиг аутентифицируется по всему
  балансировщику. Компромисс: компрометация ключа на любой ноде компрометирует
  REALITY-идентичность их всех. Это та же модель, что и у RU-моста, и её следует
  задокументировать как явное свойство безопасности.
- **Нода должна быть заранее подготовлена Ansible** — с инбаундом `api`,
  выставляющим `HandlerService` на её WireGuard-адресе, и со структурой
  инбаундов, идентичной другим нодам. Raven не настраивает Xray.

## 13. Направление на будущее (вне области)

Зрелые multi-node-панели (Marzban, Marzneshin, Remnawave) все ставят на каждую
ноду небольшой **агент**, вместо того чтобы дотягиваться до gRPC Xray напрямую
через сеть. Агент добавляет надёжность, автономность при недоступном control
plane и более чистую границу безопасности — ценой лишнего компонента, который
нужно разработать и поставлять.

Мы намеренно выбрали безагентный дизайн: на нашем масштабе, с уже имеющимися
WireGuard-сетью и Ansible, он проще и соответствует запросу в #100. Если
AlchemyLink когда-нибудь вырастет до многих нод, естественный следующий шаг —
тонкий, ограниченный по возможностям агент `raven-node` (выставляющий только
пользовательские операции, с конфигом, подтягиваемым и проверяемым по подписи из
git). Поскольку путь записи уже идёт через `core.AdminAPI`, это была бы замена
одной реализации, а не переписывание `api`/`syncer`.
